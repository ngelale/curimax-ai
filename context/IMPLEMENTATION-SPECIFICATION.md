## I. SERVICE CONTRACTS & API SPECIFICATIONS

### A. API Gateway Layer (Layer 1: Discovery)

**Endpoint Contract:**
```
GET /marketplace
GET /marketplace/search?q={query}&filters={json}
GET /marketplace/programs/{slug}
GET /marketplace/categories
```

**Rate Limiting:**
```
Per IP: 100 RPS
Per user (authenticated): 1000 RPS
Per endpoint (global): 5000 RPS

Response on limit exceeded:
  HTTP 429 Too Many Requests
  Retry-After: 60 (seconds)
  X-RateLimit-Limit: 100
  X-RateLimit-Remaining: 0
```

**Timeout Enforcement:**
```
Request timeout: 30 seconds (hard limit at API Gateway)
Database query timeout: Inherited from application
Response encoding timeout: 5 seconds
```

**Required Headers (Input):**
```
Accept: application/json
User-Agent: (browser/app identifier)
X-Request-ID: (generated by client; for deduplication)
X-Forwarded-For: (set by CDN; trust only from known IPs)
```

**Required Headers (Output):**
```
Content-Type: application/json; charset=utf-8
Cache-Control: public, max-age=600 (10 min for homepage)
ETag: (strong tag for conditional requests)
X-Request-ID: (echo client's request ID)
X-Response-Time: (milliseconds to generate response)
```

**Endpoint Contract:**
```
POST /student/register
```

```
For all POST/PUT requests:

```

**Request Validation:**
```
Schema validation: JSON Schema (version 2020-12)
Content-Type: application/json only (no form-data for APIs)
Max payload size: 50MB (for file uploads)
Timeout: 5 seconds (application logic execution)
```

**Error Response Format:**
```json
{

    "code": "VALIDATION_ERROR",

    "details": [
      {
        "field": "email",
```

## II. DATABASE SCHEMA & CONSTRAINTS

### A. Critical Tables

**applications table:**
```sql
CREATE TABLE applications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id UUID NOT NULL REFERENCES students(id),
  program_id UUID NOT NULL REFERENCES programs(id),
  
  -- State machine
  status VARCHAR(50) NOT NULL CHECK (status IN 
    ('draft', 'submitted', 'under_review', 'accepted', 'rejected', 'waitlist')),
  
  -- Idempotency
  request_id UUID UNIQUE NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  submitted_at TIMESTAMPTZ,
  
  -- Data
  personal_info JSONB,
  academic_history JSONB,
  test_scores JSONB,
  recommendations JSONB,
  essays JSONB,
  resume_url TEXT,
  
  -- Metadata
  version INT NOT NULL DEFAULT 1,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  
  INDEX (student_id, program_id),
  INDEX (program_id, status),
  INDEX (submitted_at),
  INDEX (request_id)
);

-- Partition by student_id for scaling
PARTITION BY HASH (student_id) INTO 4;
```

**student_profiles table:**
```sql
CREATE TABLE student_profiles (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL UNIQUE REFERENCES users(id),
  
  -- Personal
  first_name VARCHAR(100),
  last_name VARCHAR(100),
  email VARCHAR(255) UNIQUE,
  phone VARCHAR(20),
  
  -- Education
  current_education_level VARCHAR(50),
  interested_fields TEXT,
  
  -- Tracking
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  
  INDEX (user_id),
  INDEX (updated_at)
);
```

**programs table:**
```sql
CREATE TABLE programs (
  id UUID PRIMARY KEY,
  slug VARCHAR(255) UNIQUE NOT NULL,
  
  -- Metadata
  title VARCHAR(255) NOT NULL,
  institution_id UUID NOT NULL REFERENCES institutions(id),
  description TEXT,
  
  -- Pricing & Duration
  tuition DECIMAL(10, 2) NOT NULL,
  duration_months INT NOT NULL,
  start_date DATE,
  application_deadline DATE,
  
  --Scholarship Applicable
  is_scholarship_applicable BOOLEAN,

  -- Demand Score (precomputed hourly)
  demand_score DECIMAL(3, 1),
  demand_score_updated_at TIMESTAMPTZ,
  
  -- Metadata
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  
  INDEX (slug),
  INDEX (institution_id),
  INDEX (demand_score),
  INDEX (updated_at)
);
```

### B. Indexing Strategy

**Read-heavy tables (program discovery):**
```sql
-- Search optimization
CREATE INDEX idx_programs_full_text ON programs USING GIN(
  to_tsvector('english', title || ' ' || description)
);

-- Faceted search
CREATE INDEX idx_programs_category ON programs(category);
CREATE INDEX idx_programs_degree_level ON programs(degree_level);
CREATE INDEX idx_programs_price ON programs(tuition);

-- Sorting
CREATE INDEX idx_programs_demand ON programs(demand_score DESC);
CREATE INDEX idx_programs_rating ON programs(avg_rating DESC);
```

**Write-heavy tables (applications):**
```sql
-- Prevent duplicates
CREATE UNIQUE INDEX idx_app_student_program ON applications(
  student_id, program_id
) WHERE status != 'rejected';

-- State transitions
CREATE INDEX idx_app_status ON applications(program_id, status);

-- Audit trail
CREATE INDEX idx_app_created ON applications(created_at DESC);
```

### C. Constraints & Triggers

**Prevent concurrent application submissions:**
```sql
ALTER TABLE applications
  ADD CONSTRAINT unique_pending_app
  UNIQUE (student_id, program_id) 
  WHERE status IN ('draft', 'submitted', 'under_review');
```

**Auto-update updated_at:**
```sql
CREATE TRIGGER update_timestamp
  BEFORE UPDATE ON applications
  FOR EACH ROW
  EXECUTE FUNCTION update_timestamp_column();

CREATE FUNCTION update_timestamp_column()
  RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## III. CACHE LAYER CONFIGURATION

### A. Redis Schema

**Session Storage:**
```
Key: session:{session_id}
Value: {
  user_id: UUID,
  created_at: timestamp,
  ip_address: string,
  user_agent: string,
  permissions: [string]
}
TTL: 604800 (7 days)
Serialization: JSON
```

**Query Result Cache:**
```
Key: search:programs:{hash(query + filters)}
Value: {
  total: number,
  results: [{id, title, institution, ...}],
  facets: {...}
}
TTL: 300 (5 minutes)
Invalidation: On program.updated_at change
```

**User Saved Programs:**
```
Key: user:{user_id}:saved_programs
Value: Set of program_id
TTL: 3600 (1 hour)
Consistency: Eventually consistent
```

**Program Metadata Cache:**
```
Key: program:{program_id}
Value: {
  id, title, institution, demand_score, avg_rating, ...
}
TTL: 3600 (1 hour)
Invalidation: On program update
```

### B. Redis Cluster Configuration

**Deployment:**
```
3 nodes (each 32GB RAM, 4 CPU cores)
Replication factor: 2 (node failure tolerance)
Persistence: RDB (snapshotting every 60s) + AOF (optional)
Eviction policy: allkeys-lru (LRU for capacity)
```

**Monitoring:**
```
Metrics to track:
  - hit_rate (target > 90%)
  - evictions (should be < 10/min)
  - memory_used_percent (alert > 90%)
  - replication_lag (alert > 1s)
```

```python
intent = stripe.PaymentIntent.create(

response = {
  'client_secret': intent.client_secret,
  'payment_intent_id': intent.id
}
```

**Webhook Handler (Payment confirmation):**
```python
# POST /webhooks/stripe

@webhook_handler
def handle_payment_succeeded(event):
  intent = event['data']['object']
  app_id = intent.metadata['application_id']
  
  # Update application status
  db.applications.update(
    {'id': app_id},
    {'payment_status': 'completed', 'payment_intent_id': intent.id}
  )
  
  # Emit event
  kafka.publish('payment.processed', {
    'application_id': app_id,
    'status': 'success'
  })
```

**Webhook Signature Verification:**
```python
def verify_stripe_signature(request_body, signature_header):
  try:
    event = stripe.Webhook.construct_event(
      request_body,
      signature_header,
      vault.read('/secret/stripe/webhook_secret')
    )
    return event
```

## VIII. DEPLOYMENT PROCEDURES

### A. Database Migration Strategy

**Schema Migration Checklist:**
```
1. [ ] Write migration SQL (both up + down)
2. [ ] Test migration on staging database
3. [ ] Estimate migration time on production data
4. [ ] Plan maintenance window (if > 30s lock expected)
5. [ ] Create backup before applying
6. [ ] Apply migration (capture start/end time)
7. [ ] Verify: Run integrity checks
8. [ ] Monitor: Watch error rates for 15 minutes
9. [ ] Rollback plan documented
```

**Lock avoidance strategies:**
```
-- Avoid: ALTER TABLE ... ADD COLUMN (locks table)
-- Better: Use non-blocking migration
BEGIN;
  ALTER TABLE applications ADD COLUMN new_field DEFAULT NULL;
COMMIT;

-- Then in separate transaction, populate old data
UPDATE applications SET new_field = ...;

-- Add constraint only after populating
ALTER TABLE applications ADD CONSTRAINT ...;
```

### B. Canary Deployment

**Procedure:**
```
1. Deploy new version to Green environment (no traffic)
2. Run smoke tests: 10 critical user flows
3. Load test: 1000 concurrent users for 5 minutes
4. Shift 10% traffic → Green for 5 minutes
   - Monitor: Error rate (alert > 1%), Latency (alert > +50%)
5. Shift 25% traffic → Green for 5 minutes
6. Shift 100% traffic → Green
7. Keep Blue running for 10 minutes (instant rollback)
8. Tear down Blue after confirmed stable
```

## IX. INCIDENT RESPONSE RUNBOOKS

### Runbook 1: Search Index Down

**Detection:** Alert "Elasticsearch unavailable"

**Immediate (0-5 min):**
1. Check Elasticsearch health: `curl http://es-node:9200/_health`
2. If unhealthy, trigger circuit breaker (disable search queries)
3. Load fallback UI: "Browse by Category" (cached)
4. Page on-call
5. Capture logs: `docker logs es-container | head -200`

**Diagnosis (5-15 min):**
1. Check disk space: `df -h /es/data`
2. Check memory: `curl http://es-node:9200/_nodes/stats?human`
3. Check GC logs: `cat /es/logs/gc.log | tail -50`
4. Check Elasticsearch logs: `tail -100 /es/logs/elasticsearch.log`

**Resolution:**
- If disk full: Delete old indices; restart
- If memory pressure: Increase heap size; restart
- If corrupted: Trigger reindex from database

**Verification:**
```bash
curl -X GET "localhost:9200/_health"  # Expect: green
curl -X GET "localhost:9200/programs/_count"  # Expect: > 2000
```

### Runbook 2: Application Submission Timeout

**Detection:** P95 latency on POST /apply > 5s; Timeout errors increase

**Immediate (0-5 min):**
1. Check API logs: `tail -100 /var/log/api.log | grep -i timeout`
2. Check database load: `SELECT * FROM pg_stat_activity WHERE state != 'idle';`
3. Check application queue depth: `kafka-consumer-groups.sh --describe --group app-processor`
4. Scale up: Add 2 more API instances
5. Page on-call

**Diagnosis:**
1. Is database slow? Run `EXPLAIN ANALYZE` on slow queries
2. Is message queue full? Check Kafka lag
3. Is payment processor slow? Check Stripe dashboard
4. Is connection pool exhausted? `SELECT count(*) FROM pg_stat_activity;`

**Resolution:**
- If DB slow: Increase connection pool; add read replicas
- If Kafka lag: Scale up consumers; increase parallelism
- If payment slow: Implement deferred payment (allow submission without payment)

## XI. SECURITY HARDENING

### A. Input Validation

**All user inputs must be:**
```
1. Type-checked (string, number, array, etc.)
2. Length-bounded (max 10KB per field)
3. Format-validated (email, phone, URL, etc.)
4. Sanitized (escape HTML, SQL injection prevention)
5. Rate-limited (max X per user per hour)
```

**Example (email input):**
```python
def validate_email(email: str) -> bool:
  # Type check
  if not isinstance(email, str):
    raise TypeError("email must be string")
  
  # Length check
  if len(email) > 255:
    raise ValueError("email too long")
  
  # Format check
  if not re.match(r'^[^@]+@[^@]+\.[^@]+$', email):
    raise ValueError("invalid email format")
  
  # Sanitize (ORM handles SQL injection; no need to manually escape)
  # Rate limit
  if cache.increment(f'email_validation:{user_id}') > 100:
    raise RateLimitError("too many validations")
  
  return True
```

### B. HTTPS Enforcement

```
- Redirect HTTP → HTTPS (301 permanent)
- HSTS header: Strict-Transport-Security: max-age=31536000
- Certificate: Renewed every 60 days (30 days before expiry)
- TLS version: 1.3 minimum (disable 1.0, 1.1, 1.2)
- Cipher suite: Only modern ciphers (AEAD, ECDHE)
```

### C. CSRF Protection

```
- All POST/PUT requests require CSRF token
- Token generated per session
- Token expires with session (7 days)
- Validate token on every mutating request
- SameSite=Strict cookie flag
```
